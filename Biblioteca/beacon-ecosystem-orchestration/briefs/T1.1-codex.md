## Task

Produce the two contract templates that will govern the whole ecosystem, derived from the HarMoCAP mold, plus the copyable codec and a golden test.

### 1. `contracts/source_frame.template.json` — "Source Frame v1"

For SOURCES (HarMoCAP, MIDI drivers, ECG, EEG, simulators, audio-analysis). Rules:

- Sources emit **normalized channels** (float in declared range, polarity documented) and **never name destination instruments** (BCP principle — see INFORME_CRUZADO §D1: the source plane adopts the unified-namespace discipline).
- Namespace pattern: `/src/<source_id>/<channel>` with a `/src/<source_id>/hello` handshake. Precedent: cymatic-control's `/bridge/*`.
- **Validity states `observed | held | invalid` generalized to every channel** (from HarMoCAP `states_enum`; `held` = hold-last with decaying confidence, `invalid` = ignore value). Every channel frame carries its state.
- Stream identity (`stream_id` per process start, monotonic counters scoped by stream), presence lease + heartbeat, receiver rules (gating on contract_id match, monotonic discard, lease expiry).
- `contract_id`: SHA-256 truncated to 128 bits over the canonical JSON (sorted keys, no whitespace), excluding the id fields themselves; expected value in a golden sidecar file. Same discipline as HarMoCAP `hashes` section.
- Declared per-channel: name, range, polarity semantics, rate_hz nominal, smoothing hints ("consumer must interpolate to audio rate" where applicable).

### 2. `contracts/instrument_contract.template.json` — "Instrument Control v1"

For INSTRUMENTS (beacon-spatial, harmonic-shaper, surge-bridge, future). Rules:

- **Native namespaces are preserved** (`/beacon/*`, `/shaper/*`, ...) — do NOT flatten into a single voice model (INFORME_CRUZADO §D1 resolution: hybrid — BCP discipline on the source plane, native contracts on the instrument plane).
- Declared capabilities: address patterns with parameter placeholders (`/beacon/gain/N`), per-address: arg types, range, lag/smoothing, read/write, N bounds.
- **Bidirectional state**: how a client synchronizes on connect — `/state` dump address or periodic broadcast MUST be declared (this fixes today's unidirectional flow).
- Handshake: `/hello`-equivalent + gating rules for clients with stale contract_id.
- **Optional voice-model alias**: an optional section mapping `{gain, pan, phase}` per "voice" where the instrument naturally fits the BCP common-voice model (shaper, surge-bridge). beacon-spatial (gain/az/dist/Q/mix over 13 bands of a live analog signal) is the counterexample — the alias section must be optional, never mandatory.
- Same contract_id + golden sidecar discipline.

### 3. `src/harmonic_weaver/contract_codec.py`

Pure-stdlib codec/validator adapted from the nico-kit's `osc_codec.py`: canonical JSON serialization, contract_id computation, manifest validation (required sections present, ranges well-formed), golden sidecar check. It must be trivially copyable into other repos (no imports outside stdlib, no package-relative imports).

### 4. `tests/test_contracts_golden.py`

Golden reference test: both templates validate; contract_id of each template matches its sidecar; a mutated manifest fails the id check; a minimal valid source frame and a minimal valid instrument manifest round-trip through the codec.

### 5. `contracts/README.md`

Short usage doc: how a new source publishes Source Frame v1, how a new instrument publishes Instrument Control v1, how the weaver gates connections, versioning rules (when contract_id must bump).
