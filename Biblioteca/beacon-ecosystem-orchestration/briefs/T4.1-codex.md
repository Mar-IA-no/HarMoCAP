## Task

Write the design document that T4.2 will implement, plus the draft of the weaver's own contract. Two deliverables:

### 1. `docs/CORE_DESIGN.md`

Complete but tight (aim ~200 lines). Must specify:

- **Route data model**: a route = (source.channel → transformation chain → instrument.capability). Transformations: scale/range, curve, smoothing, gate/threshold, combiners (N sources → 1 parameter). Granularity down to a single HarMoCAP keypoint/feature of a single slot. Routes as declarative versioned presets, NOT code (this is the auditable-mapping requirement).
- **Aggregated/derived sources as first-class**: computed over N sources (e.g. "mean kinetic energy of non-focus slots" → harmonic bed). Define how aggregators are declared, their update cadence, and their validity semantics (reuse observed|held|invalid from Source Frame v1).
- **Scenes**: named bundles of routes + scene metadata; hot-swap semantics (what happens to in-flight values on switch — crossfade/held/reset policy).
- **Panic semantics**: global panic button — exact behavior per instrument class (silence, release, reset-to-defaults), state after panic, recovery path.
- **Headless server + WebSocket API**: the engine runs without UI; every UI is a thin client of the same protocol. Specify the WS API surface: subscribe state, CRUD routes, switch scene, panic, source/instrument registry events (connect/gate on contract_id), latency budget (HarMoCAP showed p50 ~7ms software — the weaver must budget its overhead and measure it).
- **Instrument registry**: instruments join by publishing their Instrument Control v1 manifest; only declared capabilities are patchable; handshake + gating on contract_id.
- **Source registry**: sources publish Source Frame v1 manifests; channels become addressable as `source_id.channel` in routes.
- **Measurement**: `reports/<run_id>/` pattern from HarMoCAP for latency/behavior evidence.

### 2. `contracts/stage.contract.draft.json`

Draft of the weaver's OWN contract-manifest (the stage plane — what UIs/clients consume), following the T1.1 template discipline: WS message types, route CRUD, scene switch, panic, registry events, state sync.
